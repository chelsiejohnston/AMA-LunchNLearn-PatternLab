Notes

1. What is a Style Guide?

Style guides can contain canonical information about:

  • Code best practices
  • Design principles
  • Content guidelines

This should not be superseded - it should be something we can build on. We don’t want to keep building new versions of this because it’s time consuming and confusing. It’s also valuable to have something that lives alongside our actual projects so that it can be helpful rather than auxiliary or unnecessary. It can also lay groundwork for future work.

2. Our Style Guide

  • Establish AMA-specific best practices for UX (examples, etc.)
  • Document guiding principles for design
  • Abstract design outside of web (sites, print, apps)
  • Document history of decisions related to UX and visual design components

Successes & Challenges

Successes

  • Concise code
  • Reveal gaps in UX/requirements from VSA - allows us to ID things that we need to simplify
  • Retrospectives, sprint planning, keeping up good rhythm
  • Pair code review is helpful

Challenges

  • Naming things is hard
  • Typography hellscape
  • Defining atoms
  • Setting the precedent is scary
  • Iterating on the precedent - choosing to table decisions with the knowledge
  • Discerning requirements from implmenetation realities (is this bad because VSA gave us weird requirements, the requirements changed, vendors made it weird, we made it weird, etc.?)
